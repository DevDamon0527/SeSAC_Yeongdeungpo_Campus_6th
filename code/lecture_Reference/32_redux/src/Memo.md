상태 관리를 체계적으로 할 수 있도록!

**Store** - 모든 상태가 저장되는 곳 -> 하나의 애플리케이션에는 하나의 스토어만!

**Action** - 상태의 변화를 일으키는 사건! => 객체로 표현! 최소한 type 속성 포함
액션은 상태를 변경하기 위한 정보를 담고 있는 객체
Q) 최소한 왜?

**Reducer** - 액션을 받아서 새로운 상태를 반환하는 함수 = Reducer는 순수 함수로, 동일한 입력에는 항상 동일한 출력을 반환해야 함
Q) 순수 함수?!

**Dispatch** - Aciton을 스토어에 전달하는 방법! - 호출되면 스토어는 해당 Action을 Reducer로 전달하여 상태 업데이트

**Selector** - 스토어에서 특정 상태를 선택하여 가져오는 함수!

저장공간(Store)

사용할 상태 선택! (Selector)

우리가 상태를 관리할 건데.

어떠한 상태의 변화를 일으키는 사건! (Action)

그 사건으로 인해 새로운 상태를 반환하는 함수 (Reducer) 동작!

(Dispacth)는 Aciton을 Store랑 연결해줌!

(Dispacth) 호출! => Store가 해당 Action을 Reducer로 전달! ==> 상태 업데이트!

**====>>>>>>> 액션 -> 스토어 -> 리듀서 => 새로운 상태 생성**
"dispatch"는 액션을 스토어에 보내는 행위를 의미

---

**Action**: 영화 대본 /
액션은 어떤 일이 일어나야 하는지 정의된 "명령"입니다.
**Reducer**: 조금 더 적합한 비유는 감독 /
감독은 대본을 바탕으로 배우들이 실제로 어떻게 행동해야 하는지 결정하고, 상황에 맞게 최종 결과물을 만들어 냅니다. 리듀서는 액션을 받아 현재 상태를 변경해 새로운 상태를 반환하는 역할을 하니까요.

**Store**: 영화 전체가 어떻게 진행되는지를 관리하는 제작사나 편집실에 비유
전체 상태(영화의 모든 장면들)를 보관하고 필요할 때마다 감독(리듀서)과 대본(액션)을 통해 상태를 갱신합니다.

**Dispatch**: 연출가나 현장 조감독 /  
Dispatch는 영화 제작 현장에서 대본을 배우들에게 전달하는 역할을 합니다.
/ 특정 상황이 발생했을 때 배우들에게 "이제 이 장면을 촬영하세요"라고 지시하는 역할을 합니다

**Selector**: Selector는 편집실에서 원하는 장면이나 정보를 찾아내는 역할

React Redux는 라이브러리

React 컴포넌트가 Redux 스토어에 쉽게 접근할 수 있도록 해주는 기능 제공

= Provider / connect / useSelector / useDispatch 훅

React 컴포넌트와 Redux를 연결

Provider - React 애플리케이션에 Redux 스토어를 제공하는 컴포넌트 (애플리케이션의 최상위 컴포넌트를 감쌈)
Q) BrowerRouter 보다도?

useSelector - 스토어에서 상태를 선택하여 React 컴포넌트로 가져옴 (Selector)

useDispatch - Action을 디스패치하기 위한 함수를 반환.

Why? 사용
컴포넌트 간의 상태 공유나 전역 상태 관리가 복잡해질 수 있음.
상태를 한 곳에서 관리하고 필요한 컴포넌트에 손쉽게 전달할 수 있게 해줌.
상태 관리 구조를 명확하게 하고, 디버깅을 용이하게 하며, 유지보수를 쉽게 해줌.

규모 가 큰 애플리케이션에서 매우 유용!
